# 슬라이드 03 — 블록체인이란?

## 블록체인 = 여러 컴퓨터가 함께 관리하는 장부

```
기존 은행/보험사                    블록체인
┌─────────────────┐              ┌────┐ ┌────┐ ┌────┐
│   중앙 서버     │              │ A  │ │ B  │ │ C  │
│  (보험사 소유)  │              │컴퓨│ │컴퓨│ │컴퓨│
│                 │              │터  │ │터  │ │터  │
│  데이터 독점    │              └────┘ └────┘ └────┘
│  임의 수정 가능 │              모두 같은 장부 공유
└─────────────────┘              한 곳이 바꾸면 나머지가 거부
```

**핵심 특징:**
- 한 번 기록된 데이터는 수정/삭제 불가
- 누구나 기록된 내용을 조회 가능
- 특정 기관이 관리하지 않음

---

### 📢 발표 스크립트

블록체인을 가장 쉽게 설명하면, "여러 컴퓨터가 함께 관리하는 장부"입니다.

기존 은행이나 보험사는 자기네 서버 한 곳에 모든 데이터를 저장합니다.
그러면 그 회사가 마음대로 데이터를 바꿀 수도 있고, 서버가 해킹되면 끝입니다.

반면 블록체인은 전 세계 수천 개의 컴퓨터가 똑같은 장부를 동시에 보관합니다.
누군가 데이터를 바꾸려면 모든 컴퓨터를 동시에 해킹해야 하는데, 이것은 사실상 불가능합니다.
그래서 "한 번 기록하면 영원히 남는다"는 특성을 가집니다.

오늘 소개할 보험 시스템도 이 블록체인 위에서 동작합니다.
보험료를 냈다, 보험금을 받았다, 이 모든 기록이 영구적으로 남습니다.

# 슬라이드 04 — 스마트 컨트랙트란?

## 스마트 컨트랙트 = 블록체인에 올라간 자동 실행 코드

```
일반 계약서                          스마트 컨트랙트
┌─────────────────────┐            ┌──────────────────────────┐
│ "조건 A가 충족되면   │            │ if (조건A == true) {      │
│  보험금을 지급한다"  │    →→→     │   지급(금액);             │
│                     │            │ }                        │
│ 사람이 판단하고      │            │ // 조건 충족되면 자동 실행 │
│ 수동으로 처리        │            │ // 사람 개입 불필요        │
└─────────────────────┘            └──────────────────────────┘
```

**핵심 특징:**
- 조건이 충족되면 **자동으로** 실행
- 코드가 공개되어 있어 **누구나 내용 확인** 가능
- 한 번 배포하면 **변경 불가** (약속이 바뀌지 않음)

---

### 📢 발표 스크립트

스마트 컨트랙트는 블록체인에 올라간 "자동 실행 코드"입니다.

일반 보험 계약서는 종이나 PDF로 되어 있죠. "이런 상황에서는 보험금을 지급한다"고 적혀 있지만,
실제로 지급하려면 담당자가 확인하고 승인하고 이체하는 수동 과정이 필요합니다.

스마트 컨트랙트는 다릅니다. "보험금 청구가 승인되면 즉시 지급한다"는 내용이
코드로 작성되어 블록체인에 올라가 있습니다.
조건이 충족되는 순간 사람의 개입 없이 자동으로 실행됩니다.

그리고 이 코드는 누구나 볼 수 있습니다.
보험사가 "우리 약관이 이렇습니다"라고 말만 하는 게 아니라,
코드로 공개되어 있어서 실제로 어떻게 동작하는지 확인할 수 있습니다.

# 슬라이드 05 — 주소(Address) 개념 ①: 지갑 주소

## 지갑 주소 = 블록체인상의 "계좌번호"

```
MetaMask 지갑
┌─────────────────────────────────────────────────────┐
│  Account #1 김덴탈                                   │
│  주소: 0x70997970C51812dc3A010C7d01b50e0d17dc79C8   │
│                                                      │
│  잔액: 1,000 USDC / ₩2,000,000 KRW                  │
└─────────────────────────────────────────────────────┘
```

**지갑 주소의 역할:**
- 토큰(돈)을 받는 주소 — 은행 계좌번호와 같음
- 트랜잭션을 보낼 때 신원 확인 수단
- 개인키(비밀번호)로만 제어 가능 → 개인키를 잃으면 접근 불가

**이 시스템의 지갑 주소:**

| 역할 | 이름 | 주소 (앞 6자리) |
|---|---|---|
| 보험사 관리자 | Account #0 | 0xf39F... |
| 피보험자 1 | 김덴탈 | 0x7099... |
| 피보험자 2 | 이치과 | 0x3C44... |
| 오라클(자동검증) | Account #3 | 0x90F7... |

---

### 📢 발표 스크립트

블록체인을 사용할 때 "주소"라는 개념이 자주 나옵니다.
주소에는 두 가지 종류가 있는데, 먼저 지갑 주소를 설명드리겠습니다.

지갑 주소는 은행의 계좌번호와 비슷합니다.
"0x7099..." 이런 식으로 생긴 42자리 문자열인데,
이 주소로 토큰, 즉 디지털 돈을 주고받습니다.

이 시스템에서는 MetaMask라는 브라우저 지갑을 사용합니다.
보험사 관리자, 피보험자 김덴탈, 이치과, 그리고 오라클 서비스 계정,
각각 고유한 지갑 주소를 가지고 있습니다.

중요한 것은 지갑 주소는 공개해도 되지만,
개인키(비밀번호에 해당)는 절대 공개하면 안 됩니다.
개인키를 알면 그 지갑의 돈을 마음대로 가져갈 수 있기 때문입니다.

# 슬라이드 06 — 주소(Address) 개념 ②: 컨트랙트 주소

## 컨트랙트 주소 = 블록체인에 배포된 "프로그램의 주소"

```
스마트 컨트랙트 배포 과정

개발자가 코드 작성              블록체인에 배포
┌────────────────┐    배포     ┌──────────────────────────────┐
│ DentalInsurance│   ─────→   │ 주소: 0xE6E340D132b5f46d...   │
│ .sol (코드)    │             │                              │
└────────────────┘             │ 이 주소로 함수 호출 가능:    │
                               │  - payPremium()  보험료 납부 │
                               │  - submitClaim() 보험금 청구 │
                               │  - requestLoan() 약관대출    │
                               └──────────────────────────────┘
```

**지갑 주소 vs 컨트랙트 주소 비교:**

| 구분 | 지갑 주소 | 컨트랙트 주소 |
|---|---|---|
| 역할 | 사람의 계좌 | 프로그램의 위치 |
| 생성 | MetaMask 생성 | 코드 배포 시 자동 생성 |
| 제어 | 개인키로 제어 | 코드 로직으로 제어 |
| 예시 | 김덴탈 0x7099... | 보험컨트랙트 0xE6E3... |
| 비유 | 내 통장 | 자판기 |

---

### 📢 발표 스크립트

두 번째 주소는 컨트랙트 주소입니다.

스마트 컨트랙트 코드를 블록체인에 배포하면,
그 프로그램이 올라간 위치를 나타내는 주소가 자동으로 생성됩니다.
이것이 컨트랙트 주소입니다.

지갑 주소와 모양은 똑같이 "0x..." 형태이지만, 역할이 다릅니다.
지갑 주소는 사람의 계좌라면, 컨트랙트 주소는 자판기 같은 겁니다.

자판기에 돈을 넣고 버튼을 누르면 음료가 나오듯이,
컨트랙트 주소로 토큰을 보내거나 함수를 호출하면
코드에 짜인 대로 자동으로 처리됩니다.

이 시스템에서 배포된 컨트랙트 주소는 4개입니다.
MockUSDC (달러 토큰), MockKRW (원화 토큰),
DentalInsurance USDC용, DentalInsurance KRW용.

각각의 주소로 해당 기능을 호출하는 방식입니다.

# 슬라이드 08 — 토큰이란?

## 토큰 = 블록체인 위에서 동작하는 디지털 화폐

```
실제 화폐와 블록체인 토큰 비교

실제 화폐                          블록체인 토큰
┌──────────────────┐             ┌──────────────────────────┐
│  한국은행 발행    │             │  스마트 컨트랙트 코드로   │
│  실물/은행 계좌   │             │  생성된 디지털 자산       │
│                  │             │                          │
│  1원 = 1원        │             │  MockKRW: 1토큰 = 1원    │
│  $1 = $1          │             │  MockUSDC: 1토큰 = $1    │
└──────────────────┘             └──────────────────────────┘
```

**이 시스템의 토큰 2종:**

| 토큰 | 이름 | 역할 | 소수점 |
|---|---|---|---|
| MockUSDC | 가짜 달러 | 달러 기반 보험료/보험금 | 6자리 ($0.000001) |
| MockKRW | 가짜 원화 | 원화 기반 보험료/보험금 | 0자리 (1토큰=1원) |

> Mock = "모의/가짜" — 실제 서비스 시 실제 스테이블코인으로 교체

---

### 📢 발표 스크립트

토큰은 블록체인 위에서 돌아가는 디지털 화폐입니다.

이 시스템에서는 두 가지 토큰을 사용합니다.
MockUSDC는 달러처럼 동작하는 토큰이고,
MockKRW는 원화처럼 동작하는 토큰입니다.

앞에 "Mock"이 붙은 이유는 테스트용이기 때문입니다.
실제 달러나 원화가 아니라, 개발 환경에서 자유롭게 발행해서 쓰는 가짜 토큰입니다.

실제 서비스로 출시한다면, MockUSDC 자리에 Circle사의 진짜 USDC를,
MockKRW 자리에 실제 원화 스테이블코인을 넣으면 됩니다.
나머지 보험 로직은 그대로 사용할 수 있도록 설계되어 있습니다.

# 슬라이드 09 — 기존 보험의 문제점

## 왜 블록체인 보험인가?

```
기존 보험 처리 과정

피보험자 청구
    ↓
서류 접수 (1~2일)
    ↓
담당자 검토 (2~5일)
    ↓          ← 이 단계에서 임의 거절 가능
심사팀 승인 (1~3일)
    ↓
지급팀 이체 (1~2일)
    ↓
입금 확인

총 소요: 평균 5~10 영업일
```

**문제점:**
- 처리 기간이 너무 길다
- 보험사가 데이터를 독점 → 투명성 없음
- 담당자 재량으로 거절 가능
- 허위 서류 제출 시 검증 어려움

---

### 📢 발표 스크립트

블록체인 보험이 왜 필요한지 이해하려면, 기존 보험의 문제점을 먼저 봐야 합니다.

현재 보험금을 청구하면 서류 접수부터 실제 입금까지 보통 5일에서 10일이 걸립니다.
그 사이에 담당자가 검토하고, 심사팀이 승인하고, 지급팀이 이체하는 과정을 거칩니다.

이 과정에서 보험사가 원하면 임의로 거절할 수도 있고,
소비자는 왜 거절됐는지 정확히 알기 어렵습니다.
또한 모든 데이터가 보험사 서버에만 있어서, 내 보험료가 얼마나 쌓였는지,
환급금이 얼마인지 보험사 말을 믿는 수밖에 없습니다.

이런 문제들을 블록체인 스마트 컨트랙트로 해결한 것이 이 시스템입니다.

# 슬라이드 10 — 블록체인 보험이 해결하는 것

## 블록체인 보험 처리 과정

```

    ↓                                  ↓  (수초)

    ↓                                  ↓

평균 5~10 영업일                   수초 내 완료
```

담당자도 없고, 심사팀도 없고, 지급팀도 없습니다.
코드가 조건을 확인하고 자동으로 처리합니다.

그리고 이 모든 과정이 블록체인에 기록되어 누구나 조회할 수 있습니다.
"내 보험금이 언제 지급됐는지", "왜 거절됐는지" 모두 투명하게 공개됩니다.

# 슬라이드 13 — 기능 ①: 청약 심사

## 가입 신청부터 증권 발급까지 자동화

```
피보험자 신청
(이름, 나이, 희망 보험료, 희망 보장한도)
        ↓
스마트 컨트랙트 자동 심사

자동 거절 조건:
├── 18세 미만 → 즉시 거절
├── 75세 초과 → 즉시 거절
├── 보장한도 / 보험료 > 100배 → 즉시 거절
└── 활성 증권 3건 이미 보유 → 즉시 거절

통과 시 → 위험점수(0~100점) 자동 산정 → Pending
        ↓
관리자 최종 승인 → 보험증권 자동 발급
```

**데모 결과:**
- 박청약 (35세): 심사 대기 → 관리자 승인 → 증권 발급
- 최이십 (20세): 심사 대기 → 관리자 승인 → 증권 발급
- 노거절 (80세): **즉시 자동 거절** (75세 초과)

---

### 📢 발표 스크립트

첫 번째 기능은 청약 심사입니다.

보험에 가입하려면 먼저 심사를 통과해야 합니다.
피보험자가 이름, 나이, 원하는 보험료, 원하는 보장한도를 입력해서 신청하면,
스마트 컨트랙트가 즉시 자동으로 심사를 진행합니다.

18세 미만이거나 75세 초과이면 즉시 거절됩니다.
보장한도가 보험료의 100배를 넘어도 거절되고,
이미 활성 증권이 3개 있어도 거절됩니다.

이 심사 기준은 코드로 공개되어 있습니다.
보험사가 "우리는 공정하게 심사합니다"라고 말만 하는 게 아니라,
코드로 기준이 명시되어 있기 때문에 자의적 차별이 불가능합니다.

자동 심사를 통과하면, 관리자가 최종 승인만 하면 증권이 자동으로 발급됩니다.

# 슬라이드 14 — 기능 ②: 보험료 자동수납

## Approve → 이후 매월 자동 이체

```
기존 방식:
매월 피보험자가 직접 → 보험료 납부 → 서명 → 처리

이 시스템:
최초 1회 [자동납부 ON] 클릭 → approve 서명 (이체 권한 부여)
        ↓
Premium Scheduler (30초 주기 폴링)
        ↓
납입 기한 도래?  잔액 충분?  자동납부 ON?
        ↓  모두 Yes
collectPremium() 자동 호출
        ↓
보험료 자동 이체 완료 → 납입 기록 블록체인에 저장
```

**핵심 개념 — approve (이체 권한 부여):**
> 피보험자가 보험 컨트랙트에게 "내 토큰을 대신 가져가도 된다"고
> 한 번만 서명해두면, 이후 자동으로 이체가 가능해짐

---

### 📢 발표 스크립트

두 번째 기능은 보험료 자동수납입니다.

기존 보험은 매월 자동이체를 설정해도 은행 계좌에서 빠져나가는 방식입니다.
이 시스템은 블록체인 토큰으로 자동이체를 구현했습니다.

핵심은 "approve"라는 개념입니다.
피보험자가 최초 한 번만 "자동납부 ON" 버튼을 클릭해서 서명하면,
보험 컨트랙트에게 "내 토큰을 매월 가져가도 된다"는 권한을 줍니다.

그 이후로는 Premium Scheduler라는 백엔드 서비스가 30초마다 확인합니다.
납입 기한이 됐는지, 잔액은 충분한지, 자동납부가 ON인지.
세 가지 조건이 모두 충족되면 보험료를 자동으로 수납합니다.

중요한 것은 이 모든 수납 기록이 블록체인에 남는다는 겁니다.
"언제 얼마를 냈는지" 영구적으로 조회 가능합니다.

# 슬라이드 15 — 기능 ③: 보험금 청구 및 오라클 검증

## 청구부터 지급까지 자동 처리

```
피보험자: 치료코드 + 금액 입력 → submitClaim()
        ↓
블록체인: ClaimSubmitted 이벤트 기록
        ↓
Oracle Service: 이벤트 감지 (즉시)
        ↓
병원 DB 조회 (Mock 또는 실제 HIRA API)
        ↓
┌─────────────────────────────────────────┐
│ VERIFIED        → 승인 + 즉시 지급      │
│ AMOUNT_EXCEEDED → 금액 초과 → 거절      │
│ UNKNOWN_CODE    → 미등록 코드 → 거절    │
└─────────────────────────────────────────┘
        ↓
oracleVerifyAndProcess() 호출
→ 결과 블록체인에 영구 기록
→ UI에 오라클 뱃지 표시
```

---

### 📢 발표 스크립트

세 번째이자 가장 핵심적인 기능, 보험금 청구 및 오라클 검증입니다.

피보험자가 치료 코드와 청구 금액을 입력해서 청구를 제출하면,
블록체인에 "청구 이벤트"가 기록됩니다.

오라클 서비스가 이 이벤트를 감지하고, 병원 데이터베이스를 조회합니다.
세 가지 결과가 나올 수 있습니다.

등록된 치료 코드이고 금액도 적정하면 VERIFIED, 즉시 승인하고 보험금을 지급합니다.
금액이 허용 한도를 초과하면 AMOUNT_EXCEEDED로 거절합니다.
존재하지 않는 치료 코드이면 UNKNOWN_CODE로 거절합니다.

이 검증 과정이 자동화되어 있기 때문에 허위 청구나 과잉 청구를 방지할 수 있습니다.
그리고 모든 결과가 블록체인에 기록되어 나중에 분쟁이 생겨도 근거가 남습니다.

# 슬라이드 17 — 기능 ⑤: 만기환급 자동 지급

## 만기일이 되면 알아서 환급

```
보험 증권 생성 시 설정:
  만기일: 2026년 12월 31일
  환급률: 70%

─────────────────────────────────────────
Maturity Watcher (10초마다 확인)

  현재 시각 >= 만기일?
  만기환급금 이미 지급됨?
       ↓ Yes / No
  processMaturityRefund() 호출
       ↓
  총 납입액 × 70% 자동 지급

예시:
  총 납입액 50 USDC × 70% = 35 USDC 자동 환급
  총 납입액 ₩700,000 × 70% = ₩490,000 자동 환급
─────────────────────────────────────────
```

**테스트 도구:** `node scripts/advance-time.js 3600` → 1시간 앞당기기

---

### 📢 발표 스크립트

다섯 번째 기능은 만기환급 자동 지급입니다.

일반적으로 만기환급형 보험은 만기가 됐을 때 보험사에 연락해서 환급을 신청해야 합니다.
이 시스템은 만기일이 되면 자동으로 지급합니다.

Maturity Watcher라는 백엔드 서비스가 10초마다 모든 증권의 만기일을 확인합니다.
만기일이 지났고 아직 환급이 안 된 증권이 있으면,
processMaturityRefund 함수를 자동으로 호출해서 피보험자 지갑으로 즉시 이체합니다.

환급 조건과 금액은 스마트 컨트랙트에 기록되어 있습니다.
보험사가 나중에 "원래 약속과 다르다"고 할 수 없고,
만기 처리를 까먹거나 늦게 할 수도 없습니다.

# 슬라이드 18 — 기능 ⑥: 약관대출

## 보험을 해지하지 않고 유동성 확보

```
약관대출 구조

보험증권 총 납입액:  100 USDC
해지환급금(70%):     70 USDC
대출 한도(80%):      56 USDC   ← 최대 대출 가능 금액

대출 신청 → requestPolicyLoan()
        ↓
  즉시 지급 (수초)

이자 계산 (투명하게 공개):
  원금 × 연 5% × 경과일수 / 365

상환 → repayPolicyLoan()
     → 원금 + 이자 일시 상환
     → 보험 계속 유지
```

**강점:**
- 보험 해지 없이 긴급 자금 마련 가능
- 이자율·계산 방식이 블록체인에 공개 → 임의 변경 불가

---

### 📢 발표 스크립트

여섯 번째 기능은 약관대출입니다.

급하게 돈이 필요할 때 보험을 해지하면 손해입니다.
약관대출은 해지하지 않고 납입 보험료를 담보로 즉시 대출받는 기능입니다.

대출 한도는 해지환급금의 80%입니다.
예를 들어 총 100 USDC를 납입했고 환급률이 70%라면,
해지환급금은 70 USDC이고, 대출 한도는 그것의 80%인 56 USDC입니다.

대출 신청하면 수초 내에 지갑으로 들어옵니다.
이자는 연 5%이고, 경과 일수만큼만 계산합니다.

기존 보험사의 약관대출은 며칠이 걸리기도 하고,
이자 계산 방식도 복잡하게 느껴질 때가 있는데,
이 시스템은 이자 계산 공식이 코드로 공개되어 있어 완전히 투명합니다.

# 슬라이드 20 — 연동 방식 ①: MetaMask 지갑 연동

## 브라우저 ↔ 블록체인 연결 방법

```
연동 흐름

브라우저 (HTML/JS)
    ↓  window.ethereum 감지
MetaMask 확장프로그램
    ↓  eth_requestAccounts
사용자 계정 연결 승인
    ↓
ethers.js (JS 라이브러리)
    ↓  provider / signer 생성
블록체인 컨트랙트 호출
    ↓  MetaMask 서명 팝업
사용자 확인 클릭
    ↓
블록체인에 트랜잭션 전송
```

**핵심 라이브러리:** ethers.js v6
- 컨트랙트 주소 + ABI(함수 명세서)로 JS에서 함수 호출
- MetaMask가 개인키로 서명하여 보안 유지

---

### 📢 발표 스크립트

이제 연동 방식을 설명드리겠습니다.

사용자가 브라우저에서 버튼을 클릭하면 어떻게 블록체인에 전달될까요?

브라우저는 MetaMask라는 크롬 확장프로그램을 통해 블록체인과 통신합니다.
"보험료 납부" 버튼을 클릭하면, ethers.js라는 JavaScript 라이브러리가
컨트랙트의 함수를 호출하는 트랜잭션을 만듭니다.
MetaMask가 서명 확인 팝업을 띄우고, 사용자가 확인하면
서명된 트랜잭션이 블록체인으로 전송됩니다.

이 과정에서 개인키는 MetaMask 밖으로 절대 나가지 않습니다.
서명만 MetaMask 내부에서 이루어지기 때문에 보안이 유지됩니다.

# 슬라이드 22 — 연동 방식 ③: 오라클 서비스 연동

## 블록체인과 외부 데이터를 연결하는 오라클

```
블록체인의 한계:
  스마트 컨트랙트는 블록체인 외부 데이터를 직접 읽을 수 없음
  → 병원 시스템에 직접 접근 불가

해결책: 오라클 (Oracle)
  블록체인 ↔ 외부 세계 사이의 신뢰된 중개자

[오라클 연동 흐름]

피보험자 청구 → ClaimSubmitted 이벤트 (블록체인)
                        ↓
              Oracle Service (Node.js) 이벤트 감지
                        ↓
              HIRA API / Mock DB 조회 (외부 세계)
                        ↓
              검증 결과를 블록체인에 기록
              oracleVerifyAndProcess() 호출
                        ↓
              승인 → 자동 지급 / 거절 → 거절 사유 기록
```

---

### 📢 발표 스크립트

오라클 연동 방식을 설명드리겠습니다.

스마트 컨트랙트는 블록체인 바깥의 데이터를 직접 읽을 수 없습니다.
병원 시스템에 직접 접근해서 "이 사람이 오늘 치료를 받았나요?" 물어볼 수 없다는 뜻입니다.

이 문제를 해결하는 것이 오라클입니다.
오라클은 블록체인과 외부 세계 사이를 연결하는 신뢰된 중개자입니다.

동작 방식은 이렇습니다.
피보험자가 청구를 제출하면 블록체인에 이벤트가 기록됩니다.
Oracle Service라는 Node.js 프로그램이 이 이벤트를 감지하고,
외부 병원 데이터베이스에 진료 내역을 조회합니다.
검증 결과를 가지고 다시 블록체인의 함수를 호출해서 승인 또는 거절을 처리합니다.

이 방식으로 블록체인의 신뢰성과 외부 데이터의 실용성을 함께 활용할 수 있습니다.

# 슬라이드 23 — 연동 방식 ④: 이벤트 기반 자동화

```
이벤트 = 블록체인에서 발생하는 "알림"

스마트 컨트랙트 이벤트 목록:

      ↓
백엔드 서비스 / UI가 이벤트 구독 (listening)
      ↓

      ↓

```

### 📢 발표 스크립트

이 시스템의 자동화는 "이벤트 기반"으로 동작합니다.

스마트 컨트랙트에서 중요한 일이 발생할 때마다 이벤트를 발생시킵니다.
예를 들어 보험금 청구가 제출되면 ClaimSubmitted 이벤트,

백엔드 서비스들이 이 이벤트를 구독하고 있다가,
자신과 관련된 이벤트가 발생하면 즉시 반응합니다.

Oracle Service는 ClaimSubmitted 이벤트를 보면 자동으로 검증을 시작합니다.
UI는 PremiumPaid 이벤트를 보면 화면의 잔액을 자동으로 갱신합니다.

이 이벤트 기반 설계 덕분에 사람이 일일이 확인하고 처리하지 않아도
시스템이 알아서 돌아갑니다.

# 슬라이드 24 — 보안 설계

## 스마트 컨트랙트 보안 3가지

```
① ReentrancyGuard — 이중 출금 공격 방지

   공격자 시도:
   withdraw() 호출 → 돈 받는 중 → 다시 withdraw() 호출
                         ↓
   ReentrancyGuard: "이미 실행 중 — 재진입 차단"

② Ownable — 관리자 전용 기능 보호

   일반 사용자 → createPolicy() 호출 시도
                         ↓
   Ownable: "owner만 가능 — 거부"

③ immutable stablecoin — 토큰 주소 고정

   배포 후 토큰 교체 불가
   → 배포 후 "사실 다른 토큰으로 바꿨어요" 불가능
```

> OpenZeppelin — 전 세계 보안 감사를 통과한 검증된 라이브러리 사용

---

### 📢 발표 스크립트

스마트 컨트랙트는 한 번 배포하면 수정이 어렵기 때문에 보안이 매우 중요합니다.
세 가지 보안 장치를 적용했습니다.

첫 번째, ReentrancyGuard입니다.
2016년 이더리움 역사상 가장 큰 해킹 사건인 "The DAO 해킹"은 재진입 공격으로 발생했습니다.
돈을 받는 도중에 다시 인출 함수를 호출하는 방식인데,
ReentrancyGuard가 이 공격을 완전히 차단합니다.

두 번째, Ownable입니다.
보험증권 생성, 청구 승인 등 민감한 기능은 관리자만 실행할 수 있습니다.

세 번째, immutable입니다.
어떤 토큰을 사용할지는 배포 시 딱 한 번 정해지고 이후 변경이 불가능합니다.
나쁜 의도로 다른 토큰으로 교체하는 것을 코드 레벨에서 막습니다.

모두 OpenZeppelin이라는, 전 세계 보안 전문가들이 검증한 라이브러리를 사용했습니다.

# 슬라이드 25 — 전체 보험 플로우 정리

## 가입부터 만기까지 한눈에

```
청약 신청
(submitApplication)
    ↓
자동 심사 (연령/비율/보유건수)
    ↓
관리자 승인 → 증권 발급
(approveApplication)
    ↓
보험료 납부 (매월)
(payPremium / collectPremium 자동)
    ↓
보험금 청구
(submitClaim)
    ↓
Oracle 자동 검증 → 즉시 지급
(oracleVerifyAndProcess)
    ↓
[필요 시] 약관대출
(requestPolicyLoan → repayPolicyLoan)
    ↓
만기일 도래 → 자동 환급
(processMaturityRefund)
```

---

### 📢 발표 스크립트

지금까지 설명한 7가지 기능이 어떻게 연결되는지 전체 플로우를 보여드리겠습니다.

먼저 피보험자가 청약을 신청합니다. 자동 심사 룰을 통과하면 관리자가 최종 승인하고 증권이 발급됩니다.

이후 매월 보험료를 납부합니다. 자동납부를 설정하면 따로 신경 쓰지 않아도 됩니다.

치과 치료를 받으면 보험금을 청구합니다.
오라클이 자동으로 검증하고 수초 내에 보험금이 지급됩니다.

급하게 돈이 필요하면 약관대출로 즉시 빌리고, 여유가 생기면 갚을 수 있습니다.

보험이 만기되면 Maturity Watcher가 자동으로 감지해서 납입 보험료의 70%를 환급합니다.

이 전체 과정에서 사람이 직접 개입해야 하는 부분은 최초 가입 심사 승인 단 한 번입니다.
나머지는 모두 스마트 컨트랙트와 백엔드 서비스가 자동으로 처리합니다.

# 슬라이드 26 — 데모 ①: USDC 기본 플로우

## 실제 동작 시연 — 달러 보험

```
시연 순서

1. 관리자(Account #0)로 접속
   → 컨트랙트 잔액 $50,000 USDC 확인
   → 청약 목록: 박청약(35세) 대기중 확인

2. 박청약 청약 승인
   → [승인] 버튼 → MetaMask 서명 → 증권 자동 발급

3. 김덴탈(Account #1)로 전환
   → MetaMask에서 계정 전환
   → USDC 잔액 1,000 USDC 확인

4. 보험료 납부
   → 증권 #1 선택 → [납부] → $50 USDC 납부

5. 보험금 청구
   → 치료코드 D0120, $80 입력 → [청구]

6. 오라클 자동 처리 확인
   → 터미널: VERIFIED 로그
   → UI: Paid + 🏥 오라클승인 뱃지
```

### 📢 발표 스크립트

이제 실제 동작하는 화면을 시연해 드리겠습니다.

먼저 관리자 계정으로 접속합니다.
보험사 준비금 5만 달러가 있고, 박청약 씨가 가입 신청을 해놓은 것이 보입니다.
박청약 씨는 35세로 심사 조건을 통과했으므로 관리자가 승인합니다.

이제 피보험자 김덴탈 계정으로 전환합니다.
MetaMask에서 계정을 바꾸고 UI를 새로고침하면 김덴탈의 보험증권이 보입니다.

보험료 $50을 납부하고, 치과 정기검진 코드인 D0120으로 $80을 청구합니다.

터미널을 보면 오라클 서비스가 청구를 감지하고 검증을 진행하는 로그가 나옵니다.
D0120은 $100이 한도인데 $80은 한도 내이므로 VERIFIED가 뜨고,
UI에서 Paid 상태와 오라클 승인 뱃지를 확인할 수 있습니다.

# 슬라이드 27 — 데모 ②: 오라클 거절 시나리오

## 허위·과잉 청구 자동 차단

```

  청구 금액: $150
  결과: AMOUNT_EXCEEDED → 자동 거절

시나리오 2 — 미등록 코드 거절

  치료코드: D9999 (존재하지 않는 코드)
  결과: UNKNOWN_CODE → 자동 거절

  관리자: [Oracle Mode OFF]
  → 청구 제출 시 자동 처리 안 됨
  → 관리자가 수동으로 승인/거절
  → [Oracle Mode ON] 다시 활성화
```

### 📢 발표 스크립트

오라클이 거절하는 시나리오도 보여드리겠습니다.

첫 번째, 금액 초과입니다.
D0120 정기검진의 최대 허용 금액은 $100인데, $150으로 청구해보겠습니다.
오라클이 AMOUNT_EXCEEDED를 반환하고 자동으로 거절됩니다.

두 번째, 미등록 코드입니다.
D9999처럼 존재하지 않는 치료 코드를 입력하면,
UNKNOWN_CODE로 거절됩니다. 허위 코드로 청구하는 것을 막을 수 있습니다.

세 번째, 오라클 모드 OFF입니다.
관리자가 오라클 모드를 끄면 자동 처리가 중단되고,
관리자가 직접 청구를 검토해서 수동으로 승인 또는 거절할 수 있습니다.
이 기능은 오라클 서비스에 문제가 생겼을 때 수동으로 전환하는 비상 장치입니다.

# 슬라이드 29 — 강점 요약 및 기존 보험과 비교

## 블록체인 보험 vs 기존 보험

|---|---|---|

### 📢 발표 스크립트

보험사 서버가 해킹되거나 파산해도 내 보험 기록은 사라지지 않습니다.

# 부록 — 예상 질문 및 답변

**Q1. 실제 돈을 쓰는 건가요?**
A. 아닙니다. 이 시스템은 개발 환경(Hardhat 로컬 블록체인)에서 동작하며, MockUSDC와 MockKRW는 실제 가치가 없는 테스트 토큰입니다. 실제 서비스로 전환하려면 실제 스테이블코인과 연동해야 합니다.

**Q2. 보험사 없이 보험이 가능한가요?**
A. 스마트 컨트랙트가 보험사의 역할 중 규칙 집행과 지급을 대신합니다. 다만 준비금 관리, 초기 심사 승인 등 일부 역할은 여전히 관리자(사람)가 필요합니다.

**Q3. 오라클이 잘못된 데이터를 줄 수도 있지 않나요?**
A. 맞습니다. 이것을 "오라클 문제"라고 합니다. 실제 서비스에서는 공식 기관(HIRA 등) API를 사용하거나, 복수의 오라클이 동의해야 처리되는 방식으로 신뢰성을 높일 수 있습니다.

**Q4. 해킹 위험은 없나요?**
A. ReentrancyGuard, Ownable 등 검증된 보안 패턴을 적용했습니다. 다만 스마트 컨트랙트 배포 전 전문 보안 감사(Audit)가 실제 서비스에서는 필수입니다.

**Q5. 원화 스테이블코인이 실제로 없는데 어떻게 하나요?**
A. 현재는 MockKRW로 시뮬레이션합니다. 실제로는 원화 결제 게이트웨이(PG사)와 연동하거나, 향후 발행될 원화 스테이블코인으로 교체하면 됩니다. 컨트랙트 코드 변경 없이 토큰 주소만 바꾸면 됩니다.
